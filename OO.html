1. Introducción al Modelado Orientado a Objetos

El Modelado Orientado a Objetos (OO) se utiliza para conceptualizar el sistema propuesto en términos de clases, atributos, métodos y relaciones, facilitando un diseño modular y escalable que es esencial para la adaptabilidad del sistema del ITSB ante cambios normativos.

Se utiliza el Lenguaje Unificado de Modelado (UML) para describir la estructura estática y dinámica del sistema, priorizando la flexibilidad en la gestión de la información académica y administrativa multi-sede.

Los principales diagramas utilizados para la fase de diseño son:

  • Diagrama de Clases: Muestra la estructura estática del sistema.
  • Diagrama de Casos de Uso: Define las funcionalidades del sistema desde la perspectiva del usuario (Actores).
  • Diagrama de Secuencia: Describe la interacción dinámica entre objetos para un escenario específico (ej. Registro de Notas).

2. Diagrama de Clases

El Diagrama de Clases es la pieza central del diseño, ya que mapea directamente las entidades clave del negocio (Estudiante, Docente, Asignatura, Pago, Sede) y define cómo se relacionan entre sí a través de la herencia, composición y asociación.

Relaciones y Cardinalidades:

  • Asociación: Una Sede se asocia con múltiples Estudiantes y Docentes (1:N), reflejando la distribución geográfica del ITSB.
  • Herencia: Se utiliza una clase abstracta Persona o Usuario de la cual heredan las clases específicas (Estudiante, Docente, Administrativo), optimizando la gestión de credenciales y datos personales.
  • Composición: La entidad Registro_Academico o Inscripción está fuertemente acoplada a Estudiante y Asignatura.

3. Objetos Clave y Principios de Herencia

Para abordar la rigidez del sistema actual, el diseño OO introduce la herencia y el polimorfismo, que permiten al sistema manejar diferentes tipos de usuarios y procesos (como la validación de notas) con una lógica flexible.

Manejo de Usuarios (Herencia):

Clase Padre Clases Hijas Ventaja OO
Usuario (o Persona) Estudiante, Docente, Administrador Centraliza atributos comunes (Nombre, CI, Sede, Credenciales) y permite polimorfismo en el acceso al sistema.
Proceso_Academico Inscripción, Carga_Nota, Cierre_Semestre Permite definir un método base Ejecutar() que es implementado de forma diferente por cada proceso, facilitando la adaptación a normativas cambiantes.

Abstracción de Sede:

La clase **Sede** encapsula los datos específicos de ubicación (La Paz, Oruro, etc.) y maneja la asociación con otras clases, asegurando que el sistema sea capaz de escalar a las cinco sedes del ITSB sin duplicidad de lógica de negocio.
